Project Summary

Quick scan of the core story.

Problem

Reporting strategies were being proposed without knowing the true end-to-end delay of data arrival, risking unrealistic near real-time promises.

Solution

Measured latency across MongoDB and SQL Server, categorized delays, and pinpointed bottlenecks by source and client.

Impact

Enabled realistic reporting frequency per client and validated improvements after data-flow changes.

Users

Sales teams and remote support engineers using the monitoring dashboard.

Key Dashboard Views

Two views used to define reporting frequency and identify bottlenecks.

Data Pipeline Architecture

From operational data sources to latency analysis and monitoring.

  • Data sources included MongoDB, SQL Server alarm data (complementary tables), and GPS records.
  • Python-based extraction and processing to compute latency metrics.
  • Results exported as CSV files for flexible, client-specific analysis.
  • Power BI dashboards loaded directly from CSV folders for rapid updates.

Key Technical Highlights

Engineering choices that made the analysis actionable across clients.

End-to-end latency measurement

Computed time deltas in minutes across multiple hops and SQL Server stages.

Criticality buckets + SLA guidance

Mapped latency into decision buckets to recommend realistic reporting frequency per client.

Source bottleneck isolation

Separated delays by data source and table stage to pinpoint bottlenecks.

Operational dashboarding

Power BI drill-down views enabled sales and support to act quickly.

Role & Responsibilities

Ownership across analysis, pipeline, dashboarding, and enablement.

  • Measured delays across MongoDB and SQL Server sources.
  • Built Python extraction and processing logic to compute latency metrics.
  • Designed Power BI dashboards with criticality coloring.

Lessons learned
  • Near real-time reporting must be grounded in observed data arrival distributions.
  • Separating latency by source and processing stage makes bottlenecks actionable.
  • Simple thresholds accelerate decisions for SLAs and expectations.
  • Monitoring enables proof of improvement after client-side changes.
